1. Evaluation
Strengths
**Right-sized for token economy.** A lean Hub + lazy-loaded Spokes is the correct topology for LLM agents. It mirrors how compilers handle headers vs. translation units: declare cheap, define expensive, include on demand.
**Boot-sequence framing is correct.** Treating
project_brief.md (/c:/nbs/abible/project_brief.md)
as a boot loader (not a wiki) forces it to stay operational rather than encyclopedic.
**Separation of concerns is sound.** Strategic state (Hub) vs. tactical constraints (Spokes) maps cleanly to the two failure modes of long-running LLM work: drift on *what we're doing* vs. drift on *how we do it*.
**"Do Not Change Without Asking" is the single highest-leverage section.** This is the one item that directly prevents the most expensive class of LLM mistakes — confident regression of a hard-won decision.
Failure Points / Edge Cases
These are ranked by how often I see them bite real projects:
**Stale Hub problem.** The Hub will lie. Every milestone you forget to update it is a session that boots from a false premise. The `Save State Protocol` partially addresses this, but only if it's *triggered automatically* rather than on user request — humans forget; rituals help.
**Spoke discovery problem.** If the Hub merely "points to Spokes when necessary," the agent has to *infer* necessity from the conversation. That's fragile. A wrong inference means I either over-load context (token bloat) or under-load it (rule violation). Spokes need explicit, deterministic activation triggers, not soft suggestions.
**Context-window asymmetry.** Some sessions auto-summarize on long runs (this one already has). The Hub needs to survive summarization. If the Hub is read once at boot and never referenced again, post-summary it becomes invisible. You need a re-anchoring mechanism.
**Decision rot.** "Completed Decisions" grows unbounded. After 50 decisions, an agent starts cherry-picking which ones to honor. Decisions need expiry, supersession links, and severity tiers (architectural vs. cosmetic).
**Roadmap drift vs. reality.** "Current Status" and "Current Roadmap" will desync the moment a session goes off-script. You need a single source of truth or an explicit reconciliation step, not two parallel narratives.
**The "polite agent" failure.** When the Hub conflicts with a live user request, an LLM defaults to obeying the user. If your Hub forbids X and you ask for X, I will probably do X without flagging it. The Hub needs an enforcement clause and the agent needs an instruction to *block and ask* on conflict.
**Versioning ambiguity.** "Standardized Naming & Versioning" without a worked example is a vibe, not a spec. Show the format; don't describe it.
**No glossary.** Domain terms ("JIT", "Outer Wall", "focus-trap reclaim") accumulate fast in this project. Without a glossary, an agent re-deriving meaning each session is a recurring tax.
What's Missing from the Hub Template
Section Why
Glossary / Term Index Locks down domain vocabulary so we don't redefine "Vault" three times.
Architectural Invariants Stricter than "Do Not Change" — these are properties the system must always have (e.g., "All keyboard handlers must be cancellable", "No raw API key may ever appear in a log line"). Invariants are testable; preferences aren't.
Known Hazards / Anti-Patterns Battle scars. ("Code shadowing from focus-trap blur reclaim," "chained listeners on shared input element.") Prevents re-discovering bugs you already paid for.
Open Questions / Deferred Decisions A holding pen so unresolved trade-offs don't get silently re-decided by an agent.
Session Log (last N entries) A 5-10 item rolling list of what was done in recent sessions. Survives summarization and gives an agent immediate continuity.
Activation Map for Spokes Explicit triggers: "If task involves IndexedDB → load .rules/data-persistence.md." Removes inference from the loop.
Boot Checklist A literal sequence: "1. Read this file. 2. Confirm Tech Stack section is accurate. 3. Read latest Session Log entry. 4. State current understanding before acting." Forces grounding.
2. Refinements
A. Make the Hub Self-Enforcing
Add a top-of-file directive block that the agent reads first:
```
AGENT DIRECTIVES (read before any other section):
- On boot, recite back the Current Status in your first response.
- If a user request conflicts with "Do Not Change Without Asking" or
  "Architectural Invariants," BLOCK AND ASK. Do not silently comply.
- If the Hub contradicts itself, flag it; do not pick a winner.
- After completing any task that satisfies a roadmap item, propose a
  Save State diff before moving on.
```
Without this, the Hub is just decoration.
B. Spokes: Deterministic Activation, Not Suggestion
Spec each Spoke with a YAML-ish header so triggers are mechanical:
```
---
spoke: data-persistence
activates_when:
  - files_touched: [db.js, config.js]
  - keywords: [indexeddb, schema, migration, store]
load_priority: high
---
```
The Hub's "Activation Map" then becomes a one-line lookup, not a judgment call.
C. Tier the "Do Not Change" Section
Three severity bands, each with a different agent behavior:
**INVARIANTS** — must never violate; block and ask, no exceptions.
**DECISIONS** — settled trade-offs; may revisit only if user explicitly reopens.
**PREFERENCES** — defaults; may deviate with one-line justification.
Mixing all three under one heading is what creates the "polite agent" drift.
D. Add a Versioned Save-State Protocol
Don't just say "update this file." Specify:
When (after every completed roadmap task; after any Architectural Invariant change; never mid-task).
What format (append-only Session Log entry + targeted edits to live sections).
How the agent proposes the update (unified diff in chat *before* writing).
A version stamp on the Hub itself (`brief_version: 0.4`) so we can spot mid-session drift.
E. Create a /.rules/_index.md
A single file inside the Spokes folder that enumerates every available Spoke, its trigger, and its one-line purpose. This is what the agent actually reads to decide what to load — keeps the Hub itself short.
F. Survive Auto-Summarization
Add a section to the Hub explicitly labeled PINNED CONTEXT — RESTATE AFTER SUMMARIZATION. Contains the 5–10 facts that must never be lost: tech stack, current task, last invariant added, current version. On any post-summary turn, the agent re-reads this block before responding.
G. One Practical Constraint
Cap the Hub at ~400 lines. Past that, you've turned it back into a wiki. Anything that grows unboundedly (decisions, session log) gets a "last 20 entries; older archived to /.rules/_archive/" rule.
