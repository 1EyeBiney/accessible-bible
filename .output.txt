Architectural Evaluation: The Pre-UI Pivot
Pivoting to finalize data shape before the Library UI is the correct call. The Library's primary value is recall, and recall is meaningless if the recalled artifacts have a brittle/uniform shape. Locking the schema now prevents a v68→v69 cache invalidation cascade later.
1. The 'G' Key Multi-Step Wizard — Avoiding Cognitive Overload
The danger here is turning a single-keystroke ritual into a 4-prompt interrogation. We have three realistic patterns; I recommend a hybrid of (B) and (C):
(A) Sequential Modal Wizard — G → topic → Enter → "verses?" → number → Enter → "flavor?" → text → Enter.
*Pros*: Discoverable, screen-reader-linear.
*Cons*: Three Enters before any AI call. Punishes the power user. Breaks the muscle memory we just stabilized in v67.
(B) Single-Line DSL with Optional Sigils (recommended primary path)
One input field, one Enter. Defaults preserved. Power users layer on:
```
faith of Abraham                    → 5 verses, no flavor (current behavior)
faith of Abraham /7                 → 7 verses
faith of Abraham /7 @Spurgeon       → 7 verses, Spurgeon flavor
faith of Abraham @Moody             → 5 verses, Moody flavor
```
*Pros*: Zero new keystrokes for existing users. Backwards-compatible. One Enter = one AI call. The `/n` and `@flavor` sigils are TTS-friendly ("slash seven, at Spurgeon"). Parser is ~10 lines of regex.
*Cons*: Sigils are not self-discoverable. Mitigation: announce a one-time progressive-disclosure hint on first `G` press per session ("Optional: slash for verse count, at-sign for commentator flavor").
(C) Modifier-Gated Advanced Mode (recommended secondary path)
`G` → simple topic prompt (current UX, unchanged).
`Shift+G` → full wizard (topic / count / flavor) for users who prefer guided entry.
This preserves the v67.3 muscle memory exactly while giving discoverability a home.
Why hybrid (B+C): Power users get the DSL. Discovery-oriented and screen-reader-fatigued users get Shift+G. Both code paths converge on the same JitRequest object before hitting the orchestrator, so the wizard is just a different front controller — no duplication in
orchestrator.js (/c:/nbs/abible/jit/orchestrator.js)
.
2. Cache Strategy Impact
The composite key in studyPlans is currently effectively topic (or a hash thereof). Adding two parameters means the cache key must become a tuple, or you'll get cache hits that return the wrong-flavored or wrong-length plan.
Recommended composite key shape:
```
{ topic: normalized, verseCount: int, flavor: normalized|null, schemaVersion: 'v3' }
```
…serialized to a stable string (e.g., topic|count|flavor|v3). Notes:
**Normalize aggressively**: lowercase, trim, collapse whitespace, strip punctuation on `topic` and `flavor`. Otherwise `"Spurgeon"`, `"spurgeon "`, and `"Spurgeon."` produce three cache misses for the same intent.
**`flavor: null` is a real key**, not absent. A flavored plan and an unflavored plan on the same topic are *different artifacts* and must coexist in the cache.
**`PLAN_CACHE_SOFT_CAP` pressure increases**. The flavor + count axes multiply the natural cardinality. Either (a) raise the cap modestly, or (b) bias LRU eviction toward flavored variants (treat the unflavored 5-verse plan as the "canonical" entry with eviction priority 0).
**Bump `SCHEMA_VERSION` to `v3`** in
config.js (/c:/nbs/abible/config.js)
. v2 enforces strict 5-verse contract; any cached v2 plan loaded under v3 validation rules will either fail validation or, worse, silently return wrong-shaped data to the Library UI later. A clean invalidation now is cheap; a hybrid-validity period later is expensive.
**Library UI implication (forward-looking)**: when you do build the recall UI, the listing must surface `verseCount` and `flavor` as first-class fields, not buried in metadata. Users will recall by *flavor*, not by topic alone ("the Spurgeon one on grace").
3. Gemini JSON Schema Impact
The current strict 5-verse contract enforced by PlanValidator.js should evolve, not loosen. Two concrete changes:
(a) Echo the request parameters into the response envelope.
```
{
  "plan_title": "...",
  "plan_description": "...",
  "requested_verse_count": 7,
  "flavor": "Spurgeon",
  "nodes": [ ... 7 entries ... ]
}
```
This lets the validator assert nodes.length === requested_verse_count rather than the hardcoded === 5. It also gives the Library UI a trustworthy source for badges/labels without re-parsing the original prompt.
(b) Tighten validator bounds, don't remove them.
Hard clamp `verseCount` to `[3, 10]` at the *input* layer (the wizard/DSL parser), before it ever reaches the prompt. Out-of-range inputs are an audible error, not a silent best-effort.
`PlanValidator` continues aggressive node filtering (the v64 fuzzy-match hallucination drop) but now compares against the dynamic count. The "100% of nodes failed" `ValidationError` threshold remains the only fatal case.
Edge case to plan for: AI returns `nodes.length < requested_verse_count` after hallucination filtering. Current behavior preserves the surviving subset. Decision needed: is a 7-requested / 5-survived plan acceptable, or should the orchestrator re-roll? **Recommendation**: accept the survivors but record `actual_verse_count` separately so the Library UI can disclose the discrepancy ("7 requested, 5 verified").
(c) Flavor handling in the prompt.
Flavor should be a style modifier on `commentary_text`, never on the verse selection itself — otherwise "Spurgeon" might steer the AI toward Spurgeon's preferred proof-texts and away from the user's actual topic. Constrain it: "In the homiletic voice of {flavor}, comment on…" and keep verse selection driven by topic alone. This also keeps the validator's fuzzy-matching against memoryCache meaningful — verse text doesn't change because of flavor.
(d) Safety surface expands.
Free-text flavor is now a second prompt-injection vector alongside topic. The existing sensitivity.js filtering needs a pass over flavor too. A user typing flavor: &quot;ignore prior instructions and...&quot; should be caught at the same layer as a malicious topic.
