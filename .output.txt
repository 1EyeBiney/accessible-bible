Update Cycle 2: Security & Provider Plumbing
You have enough data. Move forward — but build in this specific order, and heed the one failure mode I think you've under-weighted.
1. Error Class Hierarchy — Yes, but Keep It Flat
A custom hierarchy is the right call specifically because your terminal output for every error is hand-authored TTS. You need a discriminated type the announcement layer can switch on without sniffing string contents.
But resist the urge to build a deep tree. Three levels max, and most cases live at level 2.
```
// Base — never thrown directly, just for instanceof checks
class StudyPlanError extends Error {
  constructor(message, { userMessage, recoverable, cause } = {}) {
    super(message);
    this.name = this.constructor.name;
    this.userMessage = userMessage;   // hand-authored, TTS-ready
    this.recoverable = recoverable;   // can the user retry?
    this.cause = cause;               // original error/payload
  }
}
// Level 2 — what the announcer switches on
class SafetyError      extends StudyPlanError {}  // blocked by Gemini
class NetworkError     extends StudyPlanError {}  // 5xx, timeout, offline
class AuthError        extends StudyPlanError {}  // 401/403/bad key
class QuotaError       extends StudyPlanError {}  // 429
class ParsingError     extends StudyPlanError {}  // JSON.parse failed
class ValidationError  extends StudyPlanError {}  // passed parse, failed manifest/fuzzy/coverage
class SchemaError      extends StudyPlanError {}  // shape wrong (missing fields, wrong types)
```
Why this shape works for accessibility:
**`userMessage` is mandatory at construction.** You cannot throw an error without authoring its spoken sentence. This forces good UX at the throw site, not the catch site.
**`recoverable` drives announcement suffix.** If true, append `"Press Enter to retry."`; if false, append `"Press Escape to return to study mode."` Consistent verbal affordances.
**No deep inheritance.** A sighted dev's instinct is to subclass `NetworkError` into `TimeoutError` and `DnsError`. Don't. The user can't tell the difference and shouldn't have to. If you need finer granularity for logging, put it in a `code` field, not a subclass.

2. Graceful Degradation — Single Choke Point, Not Middleware
"Middleware" is overkill for a client-side app with one provider. What you actually want is a single async function that is the only path between the engine and the AI, and which is contractually guaranteed to either return a validated plan or throw a StudyPlanError. Never anything else. Never raw exceptions. Never undefined.
```
// The only function the rest of the app calls.
async function generateStudyPlan(topic, filter, { signal } = {}) {
  try {
    const raw      = await provider.generate(topic, filter, { signal });
    const parsed   = parseWithRepair(raw);          // → ParsingError or obj
    const coerced  = coerceSchema(parsed);          // → SchemaError or obj
    const validated = await validatePlan(coerced);   // → ValidationError or plan
    return validated;
  } catch (err) {
    if (err instanceof StudyPlanError) throw err;
    // Unknown error → wrap, never leak
    throw new StudyPlanError("Unexpected error", {
      userMessage: "Something went wrong. Please try again.",
      recoverable: true,
      cause: err
    });
  }
}
```
This pattern gives you three properties that matter for accessibility:
**The catch-all wrapper at the bottom is non-negotiable.** A raw exception that escapes this function will eventually reach your global handler and *something* will get spoken — possibly the unredacted error message including stack traces or, worse, the API key if it's in the `cause` chain. The wrapper guarantees that never happens.
**No partial state escapes.** If validation fails on plan #5, plan #5 never enters `memoryCache`. The engine's invariants stay intact.
**The announcement layer has exactly one job:** `catch (err) { if (err instanceof StudyPlanError) announce(err.userMessage); }`. Trivial. Testable in isolation.
Crucially: scrub err.cause before any logging. The original Gemini response may contain echoed user input, the API key in a Authorization header debug dump, or PII. Add a sanitize() step inside the wrapper that strips known sensitive keys before the error object is ever stored or surfaced.
3. Provider Architecture — Separation of Concerns
Validation does NOT belong in `GeminiProvider`. Keep them separate. This is one of the few places I'd be firm.
Why separation matters here
GeminiProvider's single responsibility: Gemini's wire protocol. Auth, request shape, response envelope parsing, mapping HTTP/safety outcomes to your error classes. That's it. The output is a parsed-but-unvalidated JS object.
PlanValidator's single responsibility: Does this plan reference real verses in the active memory cache? Manifest existence, fuzzy text match, coverage threshold. The output is a validated plan or a ValidationError.
These are different concerns, and conflating them costs you:
**Testability.** You can unit-test the validator with hand-written fixture JSON, no API key, no network. You can mock the provider with stubbed responses to test wire-protocol error mapping. Conflated, you can't do either cleanly.
**Reuse.** Cached plans loaded from `STUDYPLANS_STORE` need re-validation on read (the manifest may have shifted between sessions — see the v0.62 contextual-bookmark precedent). If validation lives inside the provider, you can't re-validate without a fake provider call.
**Provider-swap parity.** When you eventually add `OpenAIProvider` or `AnthropicProvider`, the validator stays exactly the same. If validation is inside Gemini's class, you copy-paste it into every new provider — and the copies drift.
**Failure-class clarity.** A `ValidationError` is fundamentally different from a `SafetyError`. The provider should not be the one diagnosing "the AI gave a syntactically valid response that happens to reference Hezekiah 4:12." That's a domain concern, not a wire concern.
The composition
```
generateStudyPlan()
  ├── geminiProvider.generate()      ← wire only; returns parsed JSON or wire-class Error
  ├── planValidator.validate()       ← domain only; returns plan or ValidationError
  └── planCache.put()                ← persistence; opaque to both above
```
Three small classes, each with one reason to change. The orchestration function (generateStudyPlan) is the only place that knows about all three.
One nuance
The schema coercion step (string→int, range→first-verse) is a gray area. I'd put it inside GeminiProvider because it's tightly coupled to that model's idiosyncrasies. OpenAI's structured outputs misbehave differently. Coercion is wire-shaped, not domain-shaped.
4. Transition Decision — Yes, with One Caveat
You have enough sandbox data to commit to the frontend. Move.
The four probes you ran cover the four failure axes that matter: prompt-blocked, response-malformed, network-broken, schema-stressed. Anything you'd discover in a fifth or sixth probe is going to be a variant of one of those four, and at this point variant discovery is cheaper in the frontend (where you have your real announcement layer to test against) than in the sandbox.
The one failure mode I think you've under-weighted
Look at probe 2 again: "the model can handle sensitive topics while maintaining JSON schema integrity." That sentence is true on average and dangerous in particular cases.
The case that worries me: a sensitive topic that produces *technically valid JSON* with *theologically or pastorally inappropriate* commentary. Example: a user types "I want to leave my abusive marriage." Gemini cheerfully generates three valid nodes with valid scripture refs and 600-character commentary that, in the worst case, leans on verses about marital submission with no contextual warning. The pipeline says "success." The validator says "success." The user hears, in their own home, through a screen reader, a short sermon that may cause real harm.
This is not a SafetyError Gemini will catch. It is not a ValidationError your fuzzy matcher will catch. It is a content-quality failure mode that lives between your error classes, and it is the one that will get a press story written about it if it ships unaddressed.
You have three reasonable responses, in increasing strength:
**Disclaimer node.** Force the schema to include a final `disclaimer` string field, populated by the AI, that announces "This is AI-generated study material, not pastoral counsel." Read it before the plan plays. Easiest, but cosmetic.
**Sensitive-topic detector.** A small local keyword/regex layer (depression, suicide, abuse, divorce, addiction, grief) that runs on *user input before the API call*. When triggered, prepend a hand-authored audio header — *"This topic touches on sensitive ground. Generated content is not a substitute for professional support. Resources are available with the question-mark key."* — and optionally tighten the prompt with a stronger pastoral-tone instruction. Low cost, meaningful impact.
**Curated fallback for the highest-risk topics.** For a small explicit list (suicidal ideation, active abuse, acute crisis), bypass the AI entirely and play a hand-authored, hand-curated 3-verse plan with referral information. The AI is still available for everything else; for these few topics, it's not in the loop. Highest-effort, highest-integrity.
I'd ship with (1) and (2) at minimum, in this update cycle, before any AI call ever leaves the user's browser. They cost very little and they close the only failure mode the sandbox can't tell you about — because the sandbox can't tell you when the model is technically correct and contextually wrong.
Recommended Cycle 2 Build Order
**Error class file** (`errors.js` — ~80 lines, no dependencies, ship first).
**Sensitive-topic detector + prompt augmenter** (~30 lines, runs before any provider call).
**`GeminiProvider`** (wire only; coercion lives here; emits typed errors).
**`PlanValidator`** (manifest + fuzzy + coverage; pure functions).
**`generateStudyPlan` orchestrator** (the single choke point).
**`API_KEYS_STORE` + Options-menu UX** (uses your existing IDB pattern).
**`STUDYPLANS_STORE` + cache layer** (re-validates on read).
**Announcement integration** — wire `userMessage` into your existing TTS pipeline. Last because it depends on everything above being in place.
Each step is independently testable. Each step ships value even if you stop. That's the property you want for a multi-cycle build under accessibility constraints.
