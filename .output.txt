Part 1: Architecture Audit
importmap for @google/generative-ai — Verified Correct
This is the right call. Let me articulate why so the decision is documented and resilient.
The alternatives, and why each is worse:
**Bundler (Vite/esbuild/Rollup).** Adds a build step to a project whose explicit identity is "static, offline-first, GitHub Pages." Every contributor would need Node tooling just to test a one-line change. Counter to the project's ethos.
**Direct ESM URL imports** (`import { GoogleGenerativeAI } from 'https://esm.run/@google/generative-ai'`). Works, but couples your offline-first app to a third-party CDN's uptime and supply chain. One bad esm.run update = your AI feature dies silently.
**Vendored copy** (`./vendor/generative-ai.js`). Workable, but you take on update responsibility manually, and the SDK's own internal `import` statements may resolve incorrectly without an importmap anyway.
**`importmap` + vendored or pinned CDN copy.** What you have. Native browser feature, no build step, no runtime indirection through a bundler, and you control exactly which version resolves where. This is the right pattern for an unbundled ES-module app in 2026.
Two follow-ups I'd recommend before locking it in:
**Pin the version explicitly** in the importmap. `@google/generative-ai@latest` is a footgun; `@0.21.0` (or whatever your test passed on) is reproducible. Pin it, and bump deliberately.
**Add a CSP `script-src` allowlist** that names exactly the origin in your importmap. If you're loading from a CDN, that CDN goes in the allowlist and nothing else. This is the single biggest XSS-to-key-exfil mitigation you have, and it costs nothing.
If you've already vendored a local copy under ./jit/vendor/ or similar, even better — at that point connect-src only needs to allow generativelanguage.googleapis.com, and script-src &#39;self&#39; is sufficient.

Part 2: Task 2.6 Verification
The Vault — Verified, with refinements
Reusing the existing IndexedDB instance is correct. Three refinements:
1. Single-record store, not a multi-key store.
You have one user, one machine, generally one key per provider. Don't model this as a generic key-value store with arbitrary IDs — model it as a small set of named slots:
```
{
  provider: 'gemini',          // primary key
  encryptedKey: '...',          // or raw, see below
  redactedDisplay: '••••••a3F2',
  savedAt: 1714752000000,
  modelPreference: 'gemini-2.5-flash'
}
```
When you eventually support OpenAI/Anthropic, you add provider: &#39;openai&#39; as a second record. Same store, same shape. No schema migration needed.
2. The "Configured" / "Missing" / "Last 4" redaction pattern is correct. Hold this line everywhere — including
console.log (c:\BCI\VSCode-win32-x64-1.116.0\560a9dba96\resources\app\extensions\node_modules\typescript\lib\lib.dom.d.ts:42864)
, error cause chains, and the #visual-buffer. The single most likely accidental leak is a developer adding
console.log('saved key:', keyObj) (c:\BCI\VSCode-win32-x64-1.116.0\560a9dba96\resources\app\extensions\node_modules\typescript\lib\lib.dom.d.ts:42905)
during a debug session and forgetting to remove it. Add a project rule: the raw key string is read in exactly one place (vault.getKey()) and passed directly to the SDK constructor. It is never assigned to a variable that lives longer than that call stack.
3. Optional encryption — defer, but design for it.
Don't ship encryption-at-rest on day one. Friction for screen-reader users typing a passphrase outweighs the marginal security gain at v1. But: make the schema forward-compatible. The encryptedKey field can hold a plaintext key today and an AES-GCM ciphertext tomorrow with a sibling keyEnvelope field describing the wrapping. Plan the slot now; fill it later.
The Cache — Verified, with one important correction
Slugified hash of
(topic + filter) (c:\nbs\abible\keyboard.js:181)
as the primary key is mostly right, but incomplete. You also need to invalidate when scripture data shifts.
Recall v0.62: bookmarks broke when
memoryCache (c:\nbs\abible\db.js:9)
shape shifted between Bibles and books. That's the exact pattern that will burn your plan cache. A plan generated against BSB references "Romans 1:1" with snippet "Paul, a servant of Christ." Switch to YLT — same reference, different snippet, fuzzy validator may now reject it.
Use a composite cache key:
```
function buildCacheKey({ topic, filter, model, schemaVersion, manifestId }) {
  // manifestId = identifier of the active Bible/book in memoryCache
  // schemaVersion = your prompt-schema version (e.g., 'v1')
  const normalized = [
    (topic || '').toLowerCase().trim(),
    (filter || '').toLowerCase().trim(),
    model,
    schemaVersion,
    manifestId
  ].join('|');
  return slugify(normalized);    // simple hash → short alphanumeric string
}
```
This gives you the property: a plan cached against BSB does not get served when the user has switched to YLT. The cache key is contextually unique, not just lexically unique.
Also recommended: validate plans on read, not just write. Even with the composite key, manifests can drift (you ship a
bsb3.json (c:\BCI\VSCode-win32-x64-1.116.0\560a9dba96\resources\app\extensions\node_modules\typescript\lib\lib.dom.d.ts:5127)
with corrections). Cheap re-validation on cache hit catches stale plans for free. If validation fails on cache read, evict the entry and regenerate.
TTL or no TTL? I'd say no hard TTL. Plans don't go stale based on time; they go stale based on content shifts. The composite key + on-read validation handles staleness more precisely than any time window. Skip the TTL complexity.
Eviction policy. IndexedDB doesn't have automatic LRU. Add a lastAccessed field, and lazily evict the oldest entries when you exceed a soft cap (say, 100 plans). Run eviction asynchronously after a successful read so it never blocks playback.
